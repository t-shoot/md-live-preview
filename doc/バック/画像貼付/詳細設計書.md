# 画像の貼付・ドラッグ&ドロップ — 詳細設計書

## 処理フロー(モジュール/関数単位)

### 純粋関数(`src/shared/imageAssets.ts`)
- `extensionForMimeType(mimeType: string): string | undefined` — 既知の画像MIMEタイプ(png/jpeg/gif/webp/bmp/svg+xml)を拡張子にマップ。未知の型は`undefined`。
- `generateImageFileName(existingNames: ReadonlySet<string>, timestampMs: number, ext: string): string` — `image-<timestamp>.<ext>`から開始し、衝突していれば`-1`,`-2`...と順に試して一意な名前を返す。

### Webview側: `createImagePasteHandler`(`src/webview-editor/imagePasteHandler.ts`)
`EditorView.domEventHandlers({paste, drop})` を返すファクトリ関数。
- `paste`: `event.clipboardData.items` から最初の `kind==='file' && type.startsWith('image/')` のアイテムを探す。見つかれば `event.preventDefault()` し、`FileReader.readAsDataURL` でbase64化して`onImage(pos, mimeType, dataBase64, needsOwnParagraph)`を呼ぶ。
- `drop`: `event.dataTransfer.files` から最初の画像ファイルを探す。`view.posAtCoords({x,y})`でドロップ位置を求める(取得できなければ現在のカーソル位置にフォールバック)。
- 画像が見つからない場合は`false`を返し、通常のペースト/ドロップ処理に委ねる。

#### `escapeTable(state, pos): { pos, needsOwnParagraph }`(純粋関数、テスト対象)
カーソル/ドロップ位置をそのまま挿入位置として使う前に、構文木でその位置の祖先ノードを`node.parent`で辿り、`'Table'`ノードの中にいないか確認する。
- **テーブルの中だった場合**(回帰バグ修正): 挿入位置をテーブルの終端(`node.to`)に移動し、`needsOwnParagraph: true`を返す。カーソル位置にそのまま`![](...)`を挿入すると、テーブルの生テキスト(行・セル区切りの`|`構造)に紛れ込んでしまい、見た目上テーブルの列が壊れる、あるいは再度そのテーブルにカーソルを入れて再パースされた際に内容が消えたように見える、という不具合があったため。
- **テーブルの外だった場合**: 位置はそのまま、`needsOwnParagraph: false`を返す。

### ホスト側: `DocumentSyncSession.handlePasteImage`(`src/editor/documentSync.ts`)
1. `extensionForMimeType`で拡張子を決定。`undefined`なら何もしない。
2. `vscode.Uri.joinPath(this.document.uri, '..')`でドキュメントのフォルダ、さらに`assets`サブフォルダのURIを組み立てる。
3. `vscode.workspace.fs.createDirectory(assetsDir)`でフォルダを用意(既存なら何もしない)。
4. `vscode.workspace.fs.readDirectory(assetsDir)`で既存ファイル名一覧を取得(未作成等で失敗した場合は空とみなす)。
5. `generateImageFileName`で一意なファイル名を決定し、`vscode.workspace.fs.writeFile`で保存(`Buffer.from(dataBase64, 'base64')`でデコード)。
6. `vscode.WorkspaceEdit`で`![](assets/<filename>)`(`needsOwnParagraph`が真なら先頭に`\n\n`を付与)を`atPos`(`document.positionAt`で変換)に挿入し適用。
7. `this.post({type:'setCursor', pos: atPos + insertText.length})`でカーソル移動を指示。

**編集の反映経路**: このメソッドは`applyingLocalEdit`を立てない。そのため`vscode.workspace.applyEdit`後に発火する`onDidChangeTextDocument`は、既存の`handleDocumentChanged`によって通常の外部変更(`externalUpdate`)としてWebviewへ届く。新しい「挿入結果」メッセージ種別は追加していない。

### Webview側: `setCursor`ハンドリング(`src/webview-editor/main.ts`)
`onHostMessage`の`case 'setCursor'`で、`message.pos`をドキュメント長の範囲内にクランプしたうえで選択位置に設定し、スクロールする。

## データ設計(テーブル・DB定義)
該当なし(ファイルシステム上への画像ファイル保存のみ)。

## エラーハンドリング仕様
- 未知のMIMEタイプは`extensionForMimeType`が`undefined`を返し、`handlePasteImage`はその時点で処理を中断する(ファイルもドキュメントも変更しない)。
- `assets`フォルダの`readDirectory`が失敗した場合(フォルダが空/直後で未反映等)は既存ファイル名なしとみなして続行する。

## 既知の制約
- 複数の画像が同時に貼付/ドロップされた場合、最初の1枚のみを処理する。
- 画像保存(通常は数十ms程度)の完了を待つ間にユーザーが同じ位置周辺を編集した場合、挿入位置がわずかにずれる可能性がある(ベースバージョンの厳密な再検証は行っていない)。
- テーブル以外の構造(フェンスコードブロックなど)にカーソルがある場合の退避は行っていない(テーブルほど1行の破損がすぐ表示崩れに直結する構造ではないため、対象外)。

## 修正履歴(回帰バグ)

### 1. テーブルセル内への画像貼付でテーブルが壊れる
- **症状**: テーブルのセルにカーソルがある状態で画像を貼り付けると、画像記法がテーブルの行として吸い込まれ、テーブルの列がずれる。その後そのテーブルを再度編集しようとすると、直前に貼った画像が消えたように見える。
- **原因**: 挿入位置(`atPos`)がテーブルの生テキスト内の生の文字位置だったため、`![](...)`がテーブルの`|`区切り構造に混入していた。
- **修正**: `escapeTable`でテーブル内かどうかを判定し、テーブル内であれば挿入位置をテーブル終端に退避、独立した段落として挿入するように変更(上記参照)。

### 2. 貼り付けた画像が生テキストのまま表示され、画像として描画されない(既存の潜在的制約)
- **症状**: `![](assets/xxx.png)`は正しい位置に挿入されるが、生のMarkdown記法のまま表示され、画像として描画されない。
- **原因**: WebviewはVS Code独自のオリジン(`vscode-webview://...`)で動作しており、相対パスの`<img src>`をそのまま設定しても、実ファイルの場所を指すことができない。この問題は画像貼付機能に限らず、**ローカルな相対パスの画像参照は元々一度も動作していなかった**(`sample/sample.md`の既存サンプルが意図的にネットワークURLを使っていたのはこのため)。
- **修正**:
  1. `MarkdownLivePreviewProvider.ts`の`localResourceRoots`にドキュメント自身のフォルダを追加。
  2. `documentSync.ts`の`sendInit()`が、`webview.asWebviewUri()`で変換したドキュメントフォルダのURI(`baseUri`、末尾スラッシュ付き)を`init`メッセージに含めるようにした。
  3. `livePreviewPlugin.ts`に純粋関数`resolveImageSrc(src, baseUri)`を追加。絶対URL(`https:`, `data:`など、スキーム付き)はそのまま返し、相対パスは`new URL(src, baseUri)`で解決する。`ImageWidget.toDOM()`で表示用の`src`にのみ適用し、保存されるMarkdown本文(生の相対パス)には手を加えない。
  4. `main.ts`が`init`メッセージ受信時に`setImageBaseUri(message.baseUri)`を呼び、以降のレンダリングに反映する。
- **副次効果**: この修正により、画像貼付機能に限らず、ユーザーが手で書いたローカル相対パスの画像参照(`![](./photo.png)`など)も初めて正しく表示されるようになった。

## セキュリティ上の考慮
- 保存ファイル名・パスはホスト側で完全に生成する。Webviewから渡されるのは画像データ本体とMIMEタイプのみで、ファイル名やパスは一切渡されないため、パストラバーサルや任意ファイル名注入のリスクはない。
